重拾Java（六）异常、断言、日志

发表于 2019-05-25 | 更新于 2019-06-02 | 分类于 Java

异常

在 Java 程序设计语言中，异常对象都是派生于 Throwable 类的一个实例。如果 Java 中内置的异常类不能满足需求，用户可以创建自己的异常类。

异常分类

所有的异常都是由Throwable继承而来，但是在下一层立刻分解为两个分支：Error 和 Exception：

  • Error 类层次结构描述了 Java 运行时系统内部错误和资源耗尽错误。应用程序不应该抛出这种类型的对象。如果出现了这样的内部错误，除了通告给用户，并尽力使程序安全地终止之外，再也无能为力了。这种情况很少出现
  • 在设计 Java 程序时，需要关注 Exception 层次结构。这个层次结构又分解为两个分支：一个派生于 RuntimeException，运行时异常，可以不捕捉；另一个分支为可查异常，可查异常必须进行捕捉。

派生于 RuntimeException 的异常包含下面几种情况：

  • 错误的类型转换
  • 数组访问越界
  • 访问空指针

不是派生 RuntimeException 的可查异常包括

  • 试图在文件尾部后面读取数据。
  • 试图打开一个错误格式的 URL。
  • 试图根据给定的字符串查找 Class 对象，而这个字符串表示的类并不存在。

抛出异常

方法应该在其首部声明所有可能抛出的异常，这样可以从部首反应除这个方法可能抛出哪类已检查异常

捕获异常

（1）try块：负责捕获异常，一旦try中发现异常，程序的控制权将被移交给catch块中的异常处理程序。如果抛出了一个在catch中没有声明的异常类型，方法将立即退出。

  • try语句块不可以独立存在，必须与 catch 或者 finally 块同存

（2）catch块：处理异常，比如发出警告：提示、检查配置、网络连接，记录错误等。执行完catch块之后程序跳出catch块，继续执行后面的代码。

  • 编写catch块的注意事项：多个catch块处理的异常类，要按照先catch子类后catch父类的处理方式，因为会【就近处理】异常（由上自下）。

（3）finally：最终执行的代码，用于关闭和释放资源。

throws

主方法调用method1， method1调用method2， method2中打开文件，method2可能产生异常，但是method2不进行处理，而是抛出给调用他的方法(method1)处理

自定义异常类

定义的类应该包括两个构造器，一个是默认的构造器，一个是带有详细描述信息的构造器

断言

断言机制允许在测试期间向代码中插入一些检查语句。当代码发布时，这些插入的检测语句将会被自动移走。
断言使用assert关键字，有两种形式：

  • assert默认是不启用的
    可以在运行程序时用-enableassertions 或 -ea选项启用

  • assert抛出的异常AssertionError是继承自Error的
    断言失败后，JVM会抛出一个AssertionError错误，它继承自Error

日志

一般在最开始写代码的时候总是会在代码中加入一些System.out.println方法的语句来观察代码运行的情况。这样需要反复加入和修改，日志API就是为了解决这个问题。

常用日志框架

  • sf4j
  • commons-logging

  • log4j
    Log4j是Apache的一个开放源代码项目，通过使用Log4j，我们可以控制日志信息输送的目的地是控制台、文件、GUI组件，甚至是套接口服务器、NT的事件记录器、UNIX Syslog守护进程等；

  • JDK自带logging
    java.util.logging包为JDK的日志开发包。

java.util.logging

日志级别

级别 意义 等级
OFF 这个日志级别比较特殊，用来关闭日志
SEVERE 系统比较严重的错误级别，一般是无法恢复的bug 最高
WARNING 警告提示
INFO 普通信息输出
CONFIG 静态配置信息输出，如cpu,mem
FINE FINE，FINER，FINEST三个日志级别都是用来跟踪日志信息，输出信息的详细程度依次递增
FINER
FINEST 最低
ALL 所有的日志信息都输出
